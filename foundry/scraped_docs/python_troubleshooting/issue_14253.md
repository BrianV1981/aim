## The Solution / Discussion
### Response 1
I have looked into this and found that the code for parsing configurations does not actually support something being either an integer or a string. If you want to make the code consistent with the documentation at least the following steps will be needed.

1. Testing of everything below
2. _getini_ini will need to support this mixed type (https://github.com/pytest-dev/pytest/blob/main/src/_pytest/config/__init__.py currently line 1719)
3. _getini_toml will need to support this mixed type (https://github.com/pytest-dev/pytest/blob/main/src/_pytest/config/__init__.py currently line 1780)
4.  log_cli_level will need to be configured to accept this new type. This will be a modification to pytest_addoption in logging.py (https://github.com/pytest-dev/pytest/blob/main/src/_pytest/logging.py currently line 278) 

`add_option_ini("--log-cli-level", dest="log_cli_level", default=None, help="CLI logging level")` 

will become something like 

`add_option_ini("--log-cli-level", dest="log_cli_level", type="int_or_string", default=None, help="CLI logging level")`

The simpler task is to fix the documentation to match the code.

### Response 3
On further testing it turns out that both parts are right. You can use integer log levels as long as they are quoted strings. Your problem is that the TOML parser distinguishes between "real integers" and integers that are quoted as strings. So 0 is rejected, but "0" would likely work.
