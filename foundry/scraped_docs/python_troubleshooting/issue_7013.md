# Q: v2.32.5 wheel file hash doesn't match between PyPI and CI
**Source:** https://github.com/psf/requests/issues/7013

## The Problem / Request
Looking at the hashes of the [wheel in PyPI](https://pypi.org/project/requests/#requests-2.32.5-py3-none-any.whl) (and [in the GitHub release](https://github.com/psf/requests/releases/tag/v2.32.5)) and the hash of the uploaded build artefact from [the v2.32.5 publish workflow run](https://github.com/psf/requests/actions/runs/17051830790), they differ:

* PyPI: 2462f94637a34fd532264295e186976db0f5d453d1cdd31473c85a6a161affb6
* GitHub CI: 0f39b052d7ba3d55645a49019311390fc576ea7c2a594f56eb9b67a5388a6e84

I would guess that this is because when the `publish` CI job failed, the distributions were build locally and uploaded manually. In the future, I suggest fetching the build distributions from GitHub CI build artefacts, and publishing those.

### Response 2
> When you're checking the hashes, are they the ones included in the GH release or are you seeing them published elsewhere?

@nateprewitt In general, I check that the hash in PyPI matches the hash in my generated lock file.

When the wheel was published with trusted publishing and attestations, I simply go to the commit in the attestation and verify things.

When the wheel was published without trusted publishing and attestations, I check that the hash in PyPI matches the hash of the wheel generated by CI (either by looking the CI logs, or by downloading the build artefact and computing the hash locally). I then verify the commit which CI was run from. If there's no CI, I have to tediously verify the entire downloaded wheel.

I don't generally care about the files in the GitHub release because anyone can upload them from anywhere.
