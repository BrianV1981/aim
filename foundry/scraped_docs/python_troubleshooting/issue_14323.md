# Q: [Windows] subprocess.Popen with DEVNULL raises [WinError 6] when pytest captures stdin
**Source:** https://github.com/pytest-dev/pytest/issues/14323

## The Problem / Request
## Summary

`subprocess.Popen(stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL)` raises `[WinError 6] The handle is invalid` when run inside pytest on Windows, but works fine when run directly with `python`.

The root cause: pytest's capture plugin replaces `sys.stdin` with a handle that Windows cannot `DuplicateHandle()` on. When `subprocess.Popen` does not receive an explicit `stdin` parameter, it inherits the parent's stdin and calls `_make_inheritable()` → `DuplicateHandle()` on the captured handle, which fails.

## Minimal reproduction

```python
# test_repro.py
import subprocess
import sys

def test_popen_devnull_fails():
    """FAILS under pytest, PASSES when run directly."""
    p = subprocess.Popen(
        [sys.executable, "-c", "print('hello')"],
        stdout=subprocess.DEVNULL,
        stderr=subprocess.DEVNULL,
        # stdin not specified - pytest's captured stdin triggers the bug
    )
    p.wait()
    assert p.returncode == 0

def test_popen_devnull_works_with_explicit_stdin():
    """PASSES in all cases - workaround."""
    p = subprocess.Popen(
        [sys.executable, "-c", "print('hello')"],
        stdout=subprocess.DEVNULL,
        stderr=subprocess.DEVNULL,
        stdin=subprocess.PIPE,  # Explicitly providing stdin avoids the bug
    )
    p.wait()
    assert p.returncode == 0

if __name__ == "__main__":
    test_popen_devnull_fails()
    print("Direct run: PASS")
```

## Analysis

When pytest's capture plugin is active, it replaces `sys.stdin` with a capture object. On Windows, `subprocess.Popen._get_handles()` calls `msvcrt.get_osfhandle(sys.stdin.fileno())` to get the stdin handle for the child process. The captured stdin's file descriptor maps to a Windows handle that is not valid for `DuplicateHandle()`.

Adding `-s` (disable capture) makes the test pass, confirming the issue is capture-related.

## Workaround

Explicitly pass `stdin=subprocess.PIPE` to all `subprocess.Popen` calls.

### Response 2
Thanks for trying to reproduce! The key factor is **Python 3.14** — the bug is specific to how `os.dup2()` interacts with `GetStdHandle()` on Windows, and it's **flaky** due to Windows handle recycling. Here's the full analysis:

## Root Cause

On Windows, there are **two independent mechanisms** for stdin:

1. **C runtime fd table** — `os.dup2()` / `msvcrt.get_osfhandle(0)` 
2. **Windows process-level standard handles** — `GetStdHandle(STD_INPUT_HANDLE)`

When pytest's `FDCaptureBase` starts capturing fd 0, it does:

```python
self.targetfd_save = os.dup(0)          # DuplicateHandle(X) → X', save to new fd
os.dup2(self.tmpfile.fileno(), 0)       # CloseHandle(X), fd 0 → devnull handle Y
```

After this:
- `msvcrt.get_osfhandle(0)` → Y (valid, points to devnull)
- `GetStdHandle(STD_INPUT_HANDLE)` → **X (stale! was closed by dup2)**

When `subprocess.Popen(stdin=None)` runs, it uses `GetStdHandle(STD_INPUT_HANDLE)` (line 1362 in subprocess.py), gets the **closed handle X**, then calls `DuplicateHandle(X)` → `[WinError 6] The handle is invalid`.

## Why It's Flaky

Windows may **recycle closed handle values**. If handle value X gets reused for another object between the `dup2` and the `Popen` call, `DuplicateHandle` accidentally succeeds (on a completely unrelated resource). This makes the bug intermittent.

## 100% Reliable Reproduction

Run this **outside** pytest — it simulates exactly what `FDCaptureBase` does:

## Why Python 3.13 Doesn't Trigger It

The subprocess `_get_handles` code is identical between 3.13 and 3.14. The difference is likely in how the C runtime handles `os.dup2()` and `CloseHandle` in Python 3.14's updated UCRT — needs further investigation on whether 3.13 actually closed the original handle or kept it alive.

## Fix Options

**Option A (pytest fix):** In `FDCaptureBase.start()`, after `os.dup2()`, call `ctypes.windll.kernel32.SetStdHandle(STD_INPUT_HANDLE, msvcrt.get_osfhandle(0))` to keep the Windows standard handle in sync with fd 0.

**Option B (CPython fix):** In `subprocess.Popen._get_handles`, validate the handle from `GetStdHandle` before calling `_make_inheritable` — if `DuplicateHandle` fails, fall back to `CreatePipe` (same as the `None` case).

### Response 5
Thanks for the PR! I tested the approach locally on Python 3.14.3 + Windows 11 and confirmed it fixes the `start()` path.

However, `FDCaptureBase` calls `os.dup2()` in **four** places, not just `start()`:

| Method | When it fires | dup2 target |
|--------|--------------|-------------|
| `start()` | capture begins | fd → tmpfile |
| `done()` | capture ends | fd → saved original |
| `suspend()` | `capsys.readouterr()` | fd → saved original |
| `resume()` | after readouterr, capture resumes | fd → tmpfile |

Your PR only patches `start()`. If a test does `capsys.readouterr()` (which triggers `suspend` → `resume`) and then calls `subprocess.Popen`, the handle goes stale again through the unpatched paths.

Also worth noting: the same desync affects stdout (fd 1) and stderr (fd 2), not just stdin. While `Popen(stdout=subprocess.PIPE)` bypasses `GetStdHandle`, `Popen(stdout=None)` (inherit parent) would hit the same issue on those fds.

### Suggested approach

Extract a helper that maps fd 0/1/2 → `STD_*_HANDLE` and call it after every `dup2`:

```python
@staticmethod
def _sync_win32_stdhandle(targetfd: int) -> None:
    if sys.platform != "win32":
        return
    import ctypes
    import msvcrt
    fd_to_std = {0: -10, 1: -11, 2: -12}  # STD_INPUT, STD_OUTPUT, STD_ERROR
    std_id = fd_to_std.get(targetfd)
    if std_id is None:
        return
    handle = msvcrt.get_osfhandle(targetfd)
    ctypes.windll.kernel32.SetStdHandle(std_id, handle)
```

Then add `self._sync_win32_stdhandle(self.targetfd)` after the `os.dup2()` call in `start()`, `done()`, `suspend()`, and `resume()`.

I tested this full patch locally — 10 consecutive pytest runs + a stress suite (50× Popen loops, mixed capture modes, capsys-then-Popen path) all pass with zero failures.
