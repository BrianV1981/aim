# Q: Quadratic blowup when using TestCase subtests in python 3.10
**Source:** https://github.com/pytest-dev/pytest/issues/13965

## The Problem / Request
Running the following test using pytest 9.0.0 in python 3.10 leads to a quadratic blowup:
```py
from unittest import TestCase

N = 1  # 1, 2, 3, 4, ...

class TestQuadraticBlowup(TestCase):
    def test_quadratic_blowup(self) -> None:
        for i in range(1000 * N):  # multiplying by 1000 to make bigger steps
            with self.subTest():
                pass
```
For a given `N`, it takes `O(N^2)` time to complete. Here are a few samples:
- `N=10` -> 10s
- `N=20` -> 35s
- `N=30` -> 73s
- `N=40` -> 130s

When running this test using `python -m unittest test.py`, it takes less than 0.5s for `N=40`.

I traced this test a little:
- `TestCaseFunction.runtest()` is called ([unittest.py#360](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L360))
- it calls `testcase(result=self)` ([unittest.py#389](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L389)) which is `TestCase.run()` ([case.py#557](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L557))
- it constructs `outcome = _Outcome(result)` ([case.py#582](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L582)) and assigns to `self` a line later
- the actual test is executed with `self._callTestMethod(testMethod)` ([case.py#591](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L591))
- the test repeatedly calls `with self.subTest()` which is `TestCase.subTest()` ([case.py#480](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L480))
- `TestCase.subTest()` constructs `_SubTest` ([case.py#495](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L495)) 
- it enters that subtest with `with self._outcome.testPartExecutor(self._subtest, isTest=True): yield` ([case.py#497](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L497)) which is `_Outcome.testPartExecutor()` ([case.py#55](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L55))
- when the subtest finishes _successfully_, that `_Subtest` instance is appended to the `_Outcome.errors` list with `None` as error ([case.py#79](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L79))
- after all subtests finish, `TestCase.run()` calls `self._feedErrorsToResult(result, outcome.errors)` ([case.py#599](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L599))
  - note that `outcome.errors` has `1000 * N` elements
- `TestCase._feedErrorsToResult()` iterates over all those errors ([case.py#511](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L511))
  - this gives the first, outer `O(N)`
- for each error, it calls `result.addSubTest(test.test_case, test, exc_info)` ([case.py#513](https://github.com/python/cpython/blob/9524203deefb6d4ea6a502661f855961dee1af85/Lib/unittest/case.py#L513)) which is `TestCaseFunction.addSubTest()` ([unittest.py#404](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L404))
- `TestCaseFunction.addSubTest()` eventually iterates over `self.instance._outcome.errors` ([unittest.py#455](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L455)) which is the same list as `outcome.errors` above (`1000 * N` elements)
  - this gives us the second, nested `O(N)`, which in total gives us `O(N^2)` for the entire test

There might be similar issues with skipped tests ([unittest.py#450](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L450)). Another potential hot spot is [unittest.py#318](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L318).

## The Solution / Discussion
### Response 1
Interesting. From your analysis of the code, do you think there is some workaround that pytest can do on Python 3.10 only to fix it?

(I was going to suggest a workaround of disabling the subtests plugin using `-p no:subtests`, but that doesn't work currently because the unittest plugin does subtest stuff even if the subtests plugin is disabled, we might want to fix that)

### Response 2
I'm not familiar with the codebase to be able to suggest a well-thought-out solution, and I didn't see any quick and obvious workarounds.

My ideas all revolve around some kind of caching of the values `non_subtest_skip` ([unittest.py#448](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L448)) and `subtest_errors` ([unittest.py#453](https://github.com/pytest-dev/pytest/blob/9652bc714d0cae7f849d285aaa9f5a7e42e89ee3/src/_pytest/unittest.py#L453)). I crafted a [somewhat hacky solution](https://github.com/AkaZecik/pytest/commit/22f74613c8ffe7846173b7c0a7f916baa05257ea) that replaces the original `_Outcome` class with a class that caches those values, and invalidates the cache when `_Outcome.skipped` or `_Outcome.errors` is modified (either replaced completely, or some elements are removed/added/replaced). This patch makes the test with `N=100` in python 3.10 finish in roughly the same amount of time (6~8s) as in python 3.11. Precisely, it runs ~1.25 times longer, but this might be because the code performs additional checks in the first place in python 3.10.

Let me know if this kind of approach seems reasonable.
