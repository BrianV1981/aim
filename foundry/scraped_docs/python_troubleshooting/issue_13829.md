# Q: Ini option aliases
**Source:** https://github.com/pytest-dev/pytest/issues/13829

## The Problem / Request
For #13823 I'd like strictness options to have a consistent `strict_foo` naming scheme, however the option `xfail_strict` has it in reverse. I'd like to rename it to `strict_xfail`, while keeping `xfail_strict` as an alias.

To achieve this cleanly, I propose we add support for aliases to `parser.addini()`. It's pretty useful functionality for backward compatibility, and plugins may want to use it as well.

## The Solution / Discussion
### Response 1
Not sure if I like the additional churn after we already had multiple iterations of strict options changed (even with an alias, it gets confusing if people see/use different things to do the same thing).

Note how `--strict` and `--strict-*` are currently CLI options, but `xfail_strict = True` is an ini option. IMHO naming consistency between the two isn't important, because they're two different things, and already inconsistent anyways.

If anything, IMHO we should have `strict` / `strict_markers` / `strict_config` / `strict_xfail` as ini options (see https://github.com/pytest-dev/pytest/issues/13823#issuecomment-3422982167) rather than CLI flags, and then it makes sense to make the naming consistent. But if we go there, please let's do it in one step instead of doing the `xfail_strict` rename/alias in one release and the CLI -> INI move of the rest in another.

If we end up with aliasing options, IMHO there's a discussion to be had whether to deprecate the old way. Yes, it causes churn, but so does having two ways to do the same thing (especially for newcomers which might see both in different tutorials, and then they also need to know about which pytest version they use, etc.).
