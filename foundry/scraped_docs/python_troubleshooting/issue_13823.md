# Q: `--strict` deprecation was botched - what to do about it
**Source:** https://github.com/pytest-dev/pytest/issues/13823

## The Problem / Request
In pytest 6.2, we [deprecated](https://docs.pytest.org/en/stable/deprecations.html#the-strict-command-line-option) the `--strict` option. In pytest 8, we purportedly removed it, however I completely botched the removal (https://github.com/pytest-dev/pytest/commit/1f8b39ed328a395c073db603bf4063fb3ec97218), instead of removing `--strict` I removed *its deprecation*, effectively un-deprecating it.

I wonder if we should:

1. re-deprecate it and remove in pytest 10
 
2. remove it, on the idea that it did go through a deprecation period after all
 
3. go with our original plan which is to make `--strict` an "open" flag which enables all current and future `--strict-*` flags (currently `--strict-markers` and `--strict-config`). Since `--strict` currently means `--strict-markers` this will add `--strict-config` to `--strict` and document `--strict` as a proper feature.
 
4. other

I'd like to do 3 (and BTW also add ini options for `strict` and the current strict flags since I see them as project settings not per-run settings...).

(In #13757 I'm thinking it should be another strictness flag but not decided yet)

### Response 3
> I like 3 with the option to use a list for it

Interesting idea but I see a couple of downsides:

- It makes it hard to set `strict = true` and then turn some specific `strict_foo = false`. The advantage of `strict = true` over an explicit list that it automatically enables new strict options when they are added. We can add syntax like `strict = ["all", "-strict_foo"]` but I think that's undesirable.

- We won't have a `:confval:` reference for each option.

Therefore I think separate options and a boolean strict, where the specific options override `strict`, is better.

The model I have is mypy which I think works pretty well. Users who want the "automatically updating strict" behavior use `strict`, those who don't set the specific options.

### Response 9
I started implementing `strict`. After a bit of fiddling, I think the best way is to make `--strict-markers`, `--strict-config` and `--strict` literally the same as e.g. `--override-ini strict_markers=true`. That would allow us to completely forget about the CLI flags in the code itself and it mostly does the right thing. I plan to do it with a custom argparse Action.

Another thing is implementing the ini option (e.g. `strict_markers`) fallback to `strict`. The problem is that `getini` always returns a default, so it's not possible to know if `strict_markers` was explicitly set. The two options I'm considering:

1. Add a `default` parameter to `getini`. Most intuitive, but I'm afraid this will muddle the waters, with users using it over the `default` from the `addini` (or the default of the type).
2. Add an `Config.hasini(name: str) -> bool` method.

Of course I can do it by accessing private attributes but I'd like to avoid that to ensure that external plugins are able to do the same if needed.

Currently I think I'll go with 2, but let me know if you have thoughts.

### Response 11

To recap the options from above and add some new ones:

1. Add a `default` parameter to `getini`.
2. Add a `Config.hasini(name: str) -> bool` method.
3. Add a `getini_or`/`getini_fallback`/`getini_or_else` method.
4. Add a `fallback: str` parameter to `getini` (where `fallback` is another ini key, not a value).
5. Add a `fallback: str` parameter to `addini`.

One disadvantage of the `hasini` approach I currently implemented is that I hope plugins will also adopt `strict` and I think they probably won't get the fallback right with `hasini`.

Option 5 is tempting, as it's purely declarative, but it's a bit complicated (how it interacts with `default`, need to support "fallback chains" and even loops are possible...).

For 3 vs 4, I prefer 4 over 3 since I don't think it really needs a separate method.
