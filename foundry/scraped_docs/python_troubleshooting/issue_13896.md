# Q: terminalprogress causes thousands of iTerm2 alerts
**Source:** https://github.com/pytest-dev/pytest/issues/13896

## The Problem / Request
<!--

Using pytest 9.0.0 with iTerm2 3.6.5, my test suite caused thousands of alerts like this:

<img width="385" height="106" alt="Image" src="https://github.com/user-attachments/assets/a929b703-e3cf-492c-b433-39e81a257a74" />

I understand I can disable this with `-p no:terminalprogress`, but it was alarming and others might not read through the detailed (thank you!) release notes.  iTerm2 sets two environment variables: 
```
ITERM_PROFILE=Main
ITERM_SESSION_ID=w0t1p0:3DB6DF06-FE11-40C3-9A66-9E10A193A632
```

Is it too ad-hoc to detect one of these variables and default the plugin to off?

### Response 8
Okay, so `TERM` is not useful in this case, and the others are probably not passed through by tox/Docker (and possibly others). It's basically [browser user agents](https://webaim.org/blog/user-agent-string-history/) all over again!

That being said, it looks like iTerm did [fully implement support](https://github.com/gnachman/iTerm2/commit/6a163da0118d66b774230cbf430e0912e17ad080) for the progress reporting two weeks ago, which already has a pre-release with v3.6.6beta2 (and beta 3 was released yesterday).

I don't know how iTerm's release roadmap looks like, but I'd be guessing that a v3.6.6 is around the corner. Then we'll need to think about either removing the disabling workaround again entirely, or making it conditional based on `TERM_PROGRAM_VERSION` maybe.

### Response 9
I'm thinking maybe we entirely disable it on macos for now, but probably it won't help @ideallical case which is using docker.

### Response 10
Alternatively we can try detecting iTerm2 by querying it directly. I found that iTerm2 supports a custom "DSR 1337" query sequence: https://iterm2.com/utilities/it2check. But dealing with terminals this way is messy (need to set tty mode and read from stdin etc.).
