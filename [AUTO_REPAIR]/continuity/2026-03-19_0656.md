**Debt**

The main debt is architectural debt around role clarity. “Scrivener,” “session summarizer,” and the “reasoning/sentinel half of the brain” were not enforced strongly enough in code or config, so one layer could start doing another layer’s job. That needs hard boundaries, not just intent. If there is no explicit contract saying Scrivener may only extract, normalize, and format raw history, this failure will recur.

There is also verification debt. Claims of “perfectly restored,” “technically exhaustive,” and “verified” were made before the deterministic rerun produced an obviously incomplete result. The project needs a testable definition of success for memory reconstruction: expected number of sessions found, expected turn count, expected action count, and maybe spot-check assertions against known timestamps from raw logs. Right now success language outran evidence.

The March 19 memory artifact is in ambiguous state. `memory/2026-03-19.md` was said to be exhaustive after the 22-chunk run, but the later deterministic pass suggests either overwrite risk, divergent outputs, or a broken source-selection path. There is debt in provenance: no clear chain of custody for which generator produced the current file, from which raw inputs, and whether prior good output was preserved.

There is parsing/input debt in the deterministic extractor. “No new activity” implies one of a few likely failures: wrong session selection, incorrect path resolution, broken history parser, timestamp/window filtering bug, or an extractor that only understands one transcript shape and missed the actual log format. The refactor likely prioritized removing AI behavior without enough regression coverage on log ingestion.

There is naming/config debt around model identity. The exact literal `chatGPT-5.4` matters to the user, and the system already demonstrated a tendency to “correct” or normalize it. That means model names should probably be treated as opaque user-owned strings, never auto-normalized, shortened, or inferred.

Finally, there is process debt in handling large-history summarization under CLI constraints. The chunking workaround was effective, but it appears entangled with the AI-based path that the user rejected. The codebase still needs a durable non-AI strategy for large transcript reconstruction: deterministic chunking, stable merge semantics, and preservation of chronology without synthesis. Until that exists and is verified, memory generation for large sessions remains fragile.
